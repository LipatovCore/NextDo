# NextDo

NextDo - это небольшое Django-приложение для управления личными задачами. Пользователь входит в систему, видит только свои задачи, добавляет новые, отмечает выполненные и при необходимости удаляет их.

## Функциональность

- вход и выход через встроенную аутентификацию Django;
- просмотр списка задач текущего пользователя;
- создание задачи прямо на странице списка;
- переключение статуса задачи: активна / завершена;
- удаление задачи;
- скрытие завершённых задач по умолчанию с кнопкой показа;
- автоматический редирект неизвестных URL на `/tasks/`.

## Что реализовано

В проекте используется одна основная сущность - `Task`:

- `title` - название задачи, до 100 символов;
- `user` - владелец задачи;
- `is_completed` - флаг выполнения;
- `created_at` - дата создания.

Текущее поведение приложения:

- новые задачи отображаются первыми;
- все операции с задачами доступны только авторизованному пользователю;
- чужие задачи нельзя переключать или удалять, такие запросы возвращают `404`;
- публичной регистрации пользователей нет.

## Как работает интерфейс

После входа пользователь попадает на `/tasks/`. На одной странице доступны:

- форма добавления задачи;
- список задач;
- счётчики активных и завершённых задач;
- кнопка показа или скрытия завершённых задач;
- кнопки `Готово / Вернуть` и `Удалить`.

Интерфейс собран на Django templates, а показ и скрытие завершённых задач работает через небольшой встроенный JavaScript.

## Маршруты

| URL | Назначение |
| --- | --- |
| `/login/` | вход в систему |
| `/logout/` | выход из системы |
| `/tasks/` | список задач и создание новой задачи |
| `/tasks/<task_id>/toggle/` | смена статуса задачи |
| `/tasks/<task_id>/delete/` | удаление задачи |
| `/admin/` | административная панель Django |

## Переменные окружения

Скопируйте `.env.example` в `.env` и заполните значения:

```env
SECRET_KEY=your-secret-key
DEBUG=True
ALLOWED_HOSTS=localhost,127.0.0.1

POSTGRES_DB=nextdo
POSTGRES_USER=nextdo
POSTGRES_PASSWORD=your-postgres-password
POSTGRES_HOST=db
POSTGRES_PORT=5432
```

Важно:

- `DEBUG` должен быть булевым значением, которое понимает `python-decouple`, например `True` или `False`;
- проект настроен только на PostgreSQL, fallback на SQLite не предусмотрен.

## Запуск через Docker Compose

Самый простой способ поднять проект:

```bash
docker compose up --build
```

После запуска:

- контейнер `db` поднимает PostgreSQL;
- контейнер `web` применяет миграции;
- затем собираются статические файлы;
- Django запускается через Gunicorn;
- Nginx публикует приложение на порту `80`.

Открыть приложение:

```text
http://localhost
```

## Локальный запуск без Docker

Что нужно:

- Python 3.13;
- доступный PostgreSQL;
- виртуальное окружение.

Установка зависимостей:

```bash
pip install -r requirements.txt
```

Применение миграций:

```bash
cd src
python manage.py migrate
```

Создание администратора:

```bash
python manage.py createsuperuser
```

Запуск dev-сервера:

```bash
python manage.py runserver
```

Если PostgreSQL запущен локально, укажите в `.env` корректный `POSTGRES_HOST`, например `localhost`.

## Тесты

В проекте есть тесты для:

- модели `Task`;
- страницы списка задач;
- переключения статуса задачи;
- удаления задачи;
- клиентского поведения со скрытием завершённых задач.

Запуск:

```bash
cd src
python manage.py test
```

По умолчанию `manage.py test` теперь использует SQLite, поэтому для тестов не нужно поднимать PostgreSQL.

## Что покрыто тестами

- создание задач;
- значения по умолчанию;
- сортировка по дате создания;
- связь задачи с пользователем;
- редирект анонимного пользователя на страницу входа;
- отображение только своих задач;
- успешные сценарии `create / toggle / delete`;
- защита от доступа к чужим задачам.

## Ограничения текущей версии

- нет регистрации пользователей;
- нет редактирования задач;
- нет дедлайнов, приоритетов, тегов и поиска;
- нет REST API;
- нет пагинации и расширенной фильтрации.

## Что уже есть для деплоя

В репозитории уже подготовлены:

- `Dockerfile` для Django-приложения;
- `docker-compose.yml` для связки Django + PostgreSQL + Nginx;
- `nginx.conf` для reverse proxy и отдачи статики;
- `collectstatic` при старте контейнера.

Перед реальным production-деплоем стоит дополнительно продумать:

- production-настройки безопасности Django;
- безопасное хранение секретов;
- HTTPS;
- мониторинг и резервное копирование PostgreSQL.
