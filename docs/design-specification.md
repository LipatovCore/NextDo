# Техническое задание на новый дизайн NextDo

Дата подготовки: 2026-05-19

## 1. Цель редизайна

Новый дизайн нужен, чтобы превратить NextDo из небольшой страницы задач в аккуратное рабочее приложение для ежедневного управления задачами, проектами и личными финансами. Текущий интерфейс решает базовую задачу, но визуально и структурно не готов к расширению: он построен вокруг terminal/cyberpunk-стиля, использует много inline CSS в шаблонах, не имеет полноценной навигации между будущими разделами и содержит ограниченный набор состояний интерфейса.

Основные проблемы, которые нужно решить:

- текущий визуальный стиль слишком декоративный для ежедневного рабочего dashboard-приложения;
- зеленый монохромный интерфейс и monospace-типографика снижают читаемость длинных списков, форм и будущих таблиц;
- стили дублируются внутри `base.html`, `task-list.html`, `login.html`, `logout.html`, поэтому дизайн сложно поддерживать и расширять;
- навигация сейчас ограничена логотипом, именем пользователя и кнопкой выхода;
- страница `/tasks/` одновременно является единственной рабочей страницей и местом создания задач, без общей структуры приложения;
- действия изменения данных для задач сейчас выполняются через GET-ссылки, что нужно заменить на POST-формы с CSRF;
- удаление задач не имеет подтверждения;
- нет общей системы компонентов для будущих проектов, финансов, справочников, отчетов и CRUD-страниц;
- адаптивность есть точечно, но не задает единую модель поведения для будущих таблиц, фильтров и сложных форм.

Целевой пользовательский опыт:

- приложение открывается как рабочий dashboard, а не как landing page или декоративная страница;
- пользователь быстро понимает, где находятся задачи, проекты, финансы и отчеты;
- основные действия доступны без обучения: добавить задачу, создать проект, внести расход, внести доход, открыть статистику;
- интерфейс спокойно выглядит при ежедневном использовании, не перегружает пользователя эффектами и не отвлекает от данных;
- страницы, формы, списки и таблицы выглядят единообразно;
- приложение удобно на desktop, ноутбуке, планшете и телефоне;
- дизайн заранее поддерживает будущие модули без полной переделки layout.

Общий визуальный стиль: современный, чистый, рабочий, нейтральный. Основной характер интерфейса - спокойная SaaS/dashboard-среда с хорошей типографикой, ясной иерархией, умеренными акцентами и предсказуемыми состояниями элементов.

## 2. Анализ текущего интерфейса

### Текущие страницы

В проекте сейчас есть следующие пользовательские страницы:

- `/login/` - вход через стандартную Django authentication view и шаблон `registration/login.html`;
- `/logout/` - страница выхода через `registration/logout.html`;
- `/tasks/` - список задач и форма создания новой задачи в `task/task-list.html`;
- `/admin/` - стандартная административная панель Django.

Неизвестные URL обрабатываются через `handler404`, который перенаправляет пользователя на `/tasks/`.

### Текущие UI-элементы

В `base.html` сейчас реализованы:

- общий `header`;
- логотип `NextDo_`;
- пользовательское меню;
- отображение имени пользователя;
- кнопка выхода;
- ссылка входа для неавторизованного пользователя;
- контейнер основного контента.

На странице задач используются:

- центральный контейнер задач;
- панель задач с рамкой и декоративным свечением;
- заголовок `Задачи_`;
- форма добавления задачи;
- текстовое поле `title`;
- кнопка `Добавить`;
- счетчики активных и завершенных задач;
- кнопка показа/скрытия завершенных задач;
- список задач;
- псевдо-checkbox для переключения статуса;
- кнопки `Готово`, `Вернуть`, `Удалить`;
- пустое состояние;
- небольшой inline JavaScript для подсчета и скрытия завершенных задач.

На страницах входа и выхода используются:

- центральная панель;
- заголовок;
- поля логина и пароля;
- сообщение об ошибке входа;
- кнопка входа;
- кнопки повторного входа и перехода к задачам;
- декоративный system status.

### Проблемы удобства

- У задач нет фильтрации по проекту, статусу, поиску или периоду.
- Завершенные задачи скрываются только клиентским JS без общего компонента фильтров.
- Удаление задачи доступно как обычная ссылка без подтверждения.
- Toggle и delete выполняются через GET, хотя меняют данные.
- Нет breadcrumbs, контекстных действий и понятной структуры будущих разделов.
- Auth-страницы стилистически перегружены для простой операции входа.
- Нет сообщений об успешном создании, удалении или изменении задачи.
- Ошибки формы создания задачи визуально не раскрыты пользователю в общем паттерне.

### Проблемы визуальной целостности

- Весь UI завязан на один эффектный стиль: темный фон, неоновый зеленый, uppercase, letter spacing, monospace.
- Компоненты выглядят как часть терминала, а не как элементы рабочего приложения.
- Header, auth-панели и task-панель используют похожие декоративные эффекты, но не образуют масштабируемую дизайн-систему.
- Inline CSS усложняет повторное использование стилей и повышает риск хаотичных правок.
- Нет токенов цветов, отступов, радиусов, теней и типографики.

### Проблемы адаптивности

- Для текущей страницы задач есть media queries на 768px и 480px, но они решают только локальную задачу.
- Нет общего поведения для sidebar, будущих таблиц, фильтров, модальных окон и сложных финансовых форм.
- На маленьких экранах действия задач занимают много места и могут быстро перегружать список.
- Будущие финансовые таблицы и отчеты потребуют отдельного mobile-паттерна.

### Что можно сохранить

- Django templates как основной способ рендера frontend.
- Стандартную Django auth и текущую модель доступа.
- Идею персональных данных: пользователь видит только свои задачи, проекты и финансы.
- Базовые сценарии задач: создать, отметить выполненной, вернуть в активные, удалить.
- Простую структуру server-rendered форм.
- Наличие empty state для нового пользователя.
- Текущую идею счетчиков активных и завершенных задач, но в новом визуальном исполнении.

### Что лучше полностью заменить

- Terminal/cyberpunk визуальный стиль.
- Monospace как основной шрифт интерфейса.
- Inline CSS в пользовательских шаблонах.
- Header-only навигацию.
- GET-ссылки для действий, которые меняют данные.
- Удаление без подтверждения.
- Декоративные scanning/blinking animations.
- Текущие панели auth и tasks как финальный layout.
- Одностраничную структуру приложения без dashboard и sidebar.

## 3. Общая дизайн-концепция

### Общий характер интерфейса

NextDo должен выглядеть как компактное рабочее приложение: нейтральный фон, ясные блоки, плотная, но не тесная компоновка, минимум декоративных эффектов. Интерфейс должен быть удобен для регулярного использования: быстрое сканирование данных, понятные действия, устойчивое поведение на разных экранах.

Не использовать:

- неоновое свечение как основу интерфейса;
- декоративные terminal-надписи;
- постоянный uppercase для обычных подписей;
- анимации, которые не дают функциональной пользы;
- тяжелые градиенты и фоновые эффекты.

### Цветовая палитра

Базовая палитра:

- фон приложения: `#F6F7F9` или близкий светло-серый;
- поверхность карточек и панелей: `#FFFFFF`;
- основной текст: `#111827`;
- вторичный текст: `#6B7280`;
- приглушенный текст: `#9CA3AF`;
- границы: `#E5E7EB`;
- hover-фон: `#F3F4F6`;
- focus outline: насыщенный синий или индиго, например `#2563EB`;
- primary: спокойный синий `#2563EB`;
- primary hover: `#1D4ED8`;
- success: `#16A34A`;
- warning: `#D97706`;
- danger: `#DC2626`;
- info: `#0891B2`.

Правила:

- Цвет не должен быть единственным способом передать состояние.
- Финансовые доходы можно маркировать зеленым, расходы - красным или оранжевым, баланс - нейтральным с цветовым акцентом только при необходимости.
- Цветовые акценты использовать дозированно: кнопки, активные пункты меню, статусы, важные метрики.

### Типографика

Основной шрифт: системный sans-serif stack:

```css
font-family: Inter, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
```

Если Inter не подключается, интерфейс должен корректно выглядеть на системных шрифтах.

Размеры:

- базовый текст: 14-16px;
- текст в таблицах и плотных списках: 14px;
- подписи и meta-текст: 12-13px;
- заголовок страницы: 24-32px на desktop, 22-26px на mobile;
- заголовок секции: 18-20px;
- заголовок карточки: 15-17px.

Правила:

- не использовать отрицательный letter spacing;
- не масштабировать font-size от ширины viewport;
- длинные названия задач, проектов и товаров должны переноситься и не ломать layout;
- числа, суммы и даты должны быть легко сканируемыми.

### Сетка и отступы

Базовая сетка строится на шаге 4px:

- 4px - микрозазоры;
- 8px - малые отступы внутри компактных элементов;
- 12px - отступы между связанными элементами;
- 16px - базовый внутренний отступ карточек и форм;
- 24px - отступы между крупными блоками;
- 32px - отступы между секциями на desktop.

Контентная область:

- desktop: максимальная ширина контента 1200-1440px в зависимости от страницы;
- dashboard и таблицы могут занимать всю доступную ширину рабочей области;
- auth-формы должны быть узкими: 360-420px;
- формы создания/редактирования - 640-860px, если нет табличных строк.

### Карточки

Карточки использовать для:

- метрик dashboard;
- summary-блоков;
- карточек проектов;
- mobile-представления строк таблицы;
- пустых состояний;
- модальных окон.

Внешний вид:

- фон белый;
- border 1px solid `#E5E7EB`;
- border-radius 8px;
- легкая тень только для floating-слоев: modal, dropdown, popover;
- обычные карточки не должны выглядеть как тяжелые декоративные панели.

Не вкладывать карточки в карточки. Если нужно разделить контент внутри карточки, использовать секции, границы или отступы.

### Кнопки

Варианты:

- primary - основное действие страницы;
- secondary - вторичное действие;
- ghost - тихое действие в toolbar;
- danger - удаление или опасное действие;
- icon - компактные действия в таблицах и списках;
- link button - действие, визуально похожее на ссылку, но ведущее себя как кнопка.

Размеры:

- default height: 40px;
- small height: 32px;
- large height: 44-48px;
- touch target на mobile: минимум 44px.

Состояния:

- default;
- hover;
- focus-visible с четким outline;
- active;
- disabled;
- loading с индикатором и заблокированным повторным submit.

### Формы

Поля должны иметь:

- label над полем;
- понятный placeholder только как подсказку, не вместо label;
- help text, если поле требует объяснения;
- inline error под полем;
- визуальный error state на border и тексте;
- disabled/read-only state.

Формы не должны резко менять ширину при появлении ошибок. Ошибки должны занимать заранее предсказуемое место или аккуратно добавляться под полем.

### Таблицы

Таблицы использовать для:

- расходов;
- доходов;
- товаров/объектов;
- категорий;
- отчетов;
- списков проектов при большом количестве данных.

Структура:

- header с названиями колонок;
- строки высотой 44-56px;
- hover state строки;
- right-align для сумм;
- mono/tabular numbers для денежных значений при необходимости;
- actions column справа;
- empty state вместо пустого `tbody`.

На mobile основные таблицы превращать в карточки строк. Горизонтальный scroll допустим только для аналитических таблиц, где важно сохранить матричное сравнение.

### Списки

Списки задач должны быть плотнее карточек, но читабельнее таблиц:

- checkbox слева;
- основное название задачи;
- проект или meta-данные вторым рядом, если доступны;
- статус;
- действия справа на desktop;
- действия в overflow menu или отдельной строке на mobile.

### Модальные окна

Использовать для:

- подтверждения удаления;
- быстрых форм, если они не требуют длинного сценария;
- предупреждений о потере несохраненных данных.

Модалка должна содержать:

- заголовок;
- короткое описание последствий;
- primary/secondary/danger action;
- закрытие по Esc;
- фокус внутри dialog;
- возврат фокуса на исходный элемент после закрытия.

### Уведомления

Использовать toast или message banner для:

- успешного создания;
- сохранения изменений;
- удаления;
- ошибок, не связанных с конкретным полем;
- отсутствия доступа.

Уведомления должны быть короткими и полезными: "Задача создана", "Расход сохранен", "Не удалось удалить проект".

### Пустые состояния

Empty state должен объяснять, почему данных нет, и предлагать следующий шаг.

Примеры:

- для задач: "Пока нет активных задач" + кнопка "Добавить задачу";
- для проектов: "Создайте первый проект, чтобы связывать задачи и финансы";
- для финансов: "За выбранный период операций нет" + кнопки "Добавить расход" и "Добавить доход";
- для отчетов: "Недостаточно данных для отчета" + подсказка, какие операции нужны.

### Загрузка

Для server-rendered страниц основная загрузка происходит при переходе. Для будущих динамических участков использовать:

- skeleton для карточек метрик;
- skeleton rows для таблиц;
- disabled submit button при отправке формы;
- компактный spinner только внутри кнопок или небольших блоков.

### Ошибки и валидация

Ошибки делятся на:

- field errors - рядом с конкретным полем;
- form errors - сверху формы;
- page errors - отдельный блок в контенте;
- access errors - отдельное состояние "Нет доступа".

Текст ошибки должен говорить, что исправить: "Введите название задачи", "Сумма должна быть больше 0", "Выберите проект из доступных".

### Hover, focus, active

Правила:

- hover должен быть легким: фон, border или цвет текста;
- focus-visible должен быть ярким и доступным с клавиатуры;
- active должен показывать нажатие без layout shift;
- disabled элементы не должны иметь hover-эффект;
- активный пункт навигации должен отличаться фоном, цветом и/или left indicator.

## 4. Навигация и структура интерфейса

### Общая структура

После входа пользователь должен попадать на будущую страницу `/dashboard/`. До реализации dashboard допустим редирект на `/tasks/`, но дизайн должен быть спроектирован под dashboard-first модель.

Базовый app shell:

- sidebar на desktop;
- верхняя панель с названием текущей страницы, quick actions и user menu;
- основная область контента;
- mobile header с кнопкой меню;
- mobile navigation через drawer или нижнюю навигацию для ключевых разделов.

### Главное меню

Основные пункты:

- Dashboard;
- Задачи;
- Проекты;
- Финансы;
- Отчеты;
- Настройки, если появится соответствующий раздел.

Внутри финансов:

- Обзор;
- Расходы;
- Доходы;
- Товары/объекты;
- Категории;
- Отчеты.

Если sidebar не поддерживает вложенность в первой реализации, финансовые подпункты можно показывать как tabs внутри раздела `/finance/`.

### Desktop

На ширине от 1024px:

- sidebar закреплен слева;
- ширина sidebar 240-280px;
- логотип NextDo находится сверху sidebar;
- user menu можно разместить в нижней части sidebar или в topbar;
- контент занимает оставшуюся ширину;
- topbar содержит заголовок страницы, breadcrumbs и быстрые действия.

Активный пункт:

- имеет контрастный фон;
- показывает label и иконку;
- отличается от hover-состояния;
- сохраняет читаемость при collapsed sidebar.

### Mobile и маленькие экраны

На ширине до 767px:

- постоянный sidebar скрыт;
- сверху показывается compact header с названием страницы и кнопкой меню;
- navigation открывается drawer-слоем или заменяется нижней навигацией для 3-5 основных разделов;
- вторичные разделы финансов доступны через tabs, select или отдельный список внутри страницы;
- primary action может быть в верхней панели или sticky bottom bar, если не перекрывает контент.

### Breadcrumbs

Breadcrumbs нужны на вложенных страницах:

- `/projects/<id>/`;
- `/finance/expenses/new/`;
- `/finance/expenses/<id>/edit/`;
- `/finance/reports/`;
- будущие detail-страницы.

На простых верхнеуровневых страницах breadcrumbs можно не показывать.

### Быстрые действия

В topbar или page header предусмотреть быстрые действия:

- Добавить задачу;
- Создать проект;
- Добавить расход;
- Добавить доход.

На desktop быстрые действия можно показывать кнопками. На mobile при нехватке места использовать одну primary button и overflow menu.

## 5. Страницы проекта

### Login: `/login/`

Назначение: вход пользователя в систему через стандартную Django auth.

Основные сценарии:

- пользователь вводит логин и пароль;
- пользователь видит ошибку при неверных данных;
- после успешного входа пользователь попадает на `/dashboard/` или временно на `/tasks/`.

Данные:

- username;
- password;
- hidden `next`.

Действия:

- войти;
- при необходимости перейти к восстановлению доступа, если такая функция будет добавлена позже.

Блоки:

- логотип NextDo;
- заголовок "Вход";
- поля логина и пароля;
- form-level error;
- submit button.

Внешний вид:

## 6. Компоненты дизайна

### Button

Назначение: запуск действий.

Где используется: все формы, page headers, таблицы, модалки, auth-страницы.

Внешний вид:

- border-radius 6-8px;
- высота 40px по умолчанию;
- горизонтальный padding 12-16px;
- текст без uppercase по умолчанию;
- иконка слева допустима для частых действий.

Состояния:

- default;
- hover;
- focus-visible;
- active;
- disabled;
- loading.

Варианты:

- primary;
- secondary;
- ghost;
- danger;
- icon-only;
- link.

Правила:

- primary action на странице обычно один;
- danger использовать только для необратимых действий;
- loading button не должен позволять повторную отправку;
- icon-only button обязательно имеет `aria-label` или tooltip.

### Input

Назначение: ввод короткого текста, суммы, даты, поиска.

Где используется: login, task create, filters, finance forms, project forms.

Внешний вид:

- высота 40px;
- border `#D1D5DB`;
- фон белый;
- focus-visible outline или border primary;
- placeholder приглушенный.

Состояния:

- default;
- hover;
- focus;
- disabled;
- read-only;
- error;
- success, если нужна явная валидация.

Варианты:

- text;
- password;
- search;
- date;
- number/amount.

Правила:

- label обязателен;
- placeholder не заменяет label;
- для сумм показывать единицу `RUB` или формат в help text;
- ошибки показывать под полем.

### Select/Dropdown

Назначение: выбор проекта, категории, статуса, периода, товара/объекта.

Где используется: filters, finance forms, task project binding, reports.

Внешний вид:

- такой же размер и border, как у input;
- стрелка справа;
- выбранное значение должно быть читаемым;
- длинные значения обрезать с tooltip или переносить в custom dropdown, если потребуется.

Состояния:

- default;
- hover;
- focus;
- disabled;
- error;
- open.

Варианты:

- native select для простых списков;
- searchable select для товаров/объектов и длинных справочников в будущем.

Правила:

- не использовать custom dropdown без необходимости;
- пустой option должен ясно говорить "Все проекты" или "Выберите проект";
- в формах показывать ошибку при обязательном выборе.

### Checkbox/Radio

Назначение: бинарный выбор или выбор одного значения из набора.

Где используется: task completion, filters, settings, form options.

Внешний вид:

- стандартная форма с улучшенным focus;
- checkbox 18-20px;
- label кликабелен;
- checked state виден не только цветом.

Состояния:

- unchecked;
- checked;
- indeterminate, если понадобится;
- focus;
- disabled;
- error.

Варианты:

- checkbox;
- radio;
- switch только для настроек, где включается постоянное поведение.

Правила:

- для завершения задачи checkbox должен отправлять POST или быть частью POST-action;
- не использовать декоративный `a` как checkbox.

### Textarea

Назначение: ввод описания проекта, заметки к доходу/расходу, комментариев.

Где используется: project form, expense form, income form.

Внешний вид:

- min-height 96-120px;
- resize vertical;
- общий стиль input.

Состояния:

- default;
- focus;
- disabled;
- error.

Варианты:

- compact note;
- full description.

Правила:

- не использовать для коротких названий;
- длинный текст не должен ломать карточки и таблицы.

### Table

Назначение: структурированный вывод списков и отчетов.

Где используется: finance expenses, incomes, items, reports, projects при большом количестве.

Внешний вид:

- header row с нейтральным фоном;
- row separators;
- hover row;
- compact actions справа;
- суммы right-aligned.

Состояния:

- default;
- hover;
- selected, если появится массовый выбор;
- loading skeleton;
- empty;
- error.

Варианты:

- data table;
- compact table;
- report table.

Правила:

- на mobile заменять на card rows для CRUD-списков;
- не прятать критичные данные в горизонтальный scroll;
- action column должна оставаться предсказуемой.

### Card

Назначение: группировка данных или метрик.

Где используется: dashboard, project cards, metric cards, empty states, mobile row cards.

Внешний вид:

- white surface;
- subtle border;
- radius 8px;
- padding 16-24px;
- без тяжелых теней.

Состояния:

- default;
- hover only для clickable cards;
- focus-visible for links/cards;
- selected, если карточка выбирается.

Варианты:

- metric card;
- content card;
- empty card;
- row card.

Правила:

- не вкладывать карточки в карточки;
- clickable card должна иметь ясный focus state.

### Badge/Status Label

Назначение: показать статус, категорию, тип операции, проект.

Где используется: tasks, projects, finance tables, reports.

Внешний вид:

- small pill или compact label;
- font-size 12-13px;
- background light;
- border или text color по смыслу.

Состояния:

- active;
- completed;
- archived;
- income;
- expense;
- category;
- project.

Варианты:

- neutral;
- success;
- warning;
- danger;
- info.

Правила:

- badge не заменяет полноценный текст действия;
- не использовать слишком много цветов на одной странице.

### Alert/Notification

Назначение: сообщить об ошибке, успехе или важном предупреждении.

Где используется: формы, page states, после CRUD-действий.

Внешний вид:

- inline alert для формы/страницы;
- toast для кратких success-событий;
- иконка + текст;
- цвет соответствует типу.

Состояния:

- info;
- success;
- warning;
- error.

Варианты:

- page alert;
- form alert;
- toast.

Правила:

- success toast должен исчезать сам;
- error alert остается, пока пользователь не исправит проблему или не закроет сообщение;
- текст должен быть конкретным.

### Modal/Dialog

Назначение: подтверждение опасных действий и краткие блокирующие сценарии.

Где используется: удаление задач, проектов, финансовых операций, предупреждение о несохраненных изменениях.

Внешний вид:

- centered dialog;
- backdrop;
- width 360-560px;
- header, body, footer actions.

Состояния:

- open;
- closing;
- loading while action;
- error inside modal.

Варианты:

- confirm;
- danger confirm;
- simple form dialog.

Правила:

- удаление всегда требует явного подтверждения;
- Esc и close button должны работать, если действие не выполняется;
- фокус должен оставаться внутри dialog.

### Layout Containers

Назначение: управлять шириной и структурой страниц.

Где используется: app shell, page content, form pages, dashboard.

Внешний вид:

- `.app-shell`;
- `.sidebar`;
- `.topbar`;
- `.page`;
- `.page-header`;
- `.content-grid`;
- `.form-container`.

Состояния:

- sidebar expanded;
- sidebar collapsed;
- mobile drawer open.

Варианты:

- dashboard layout;
- list layout;
- form layout;
- auth layout.

Правила:

- layout классы не должны зависеть от конкретной модели данных;
- ширина и отступы задаются системно, а не случайными значениями на каждой странице.

### Sidebar/Header

Назначение: основная навигация и контекст страницы.

Где используется: все авторизованные страницы.

Внешний вид:

- sidebar на desktop;
- compact header на mobile;
- активный пункт с фоном и индикатором;
- user menu доступно, но не доминирует.

Состояния:

- active item;
- hover item;
- focus item;
- collapsed/open mobile nav.

Варианты:

- desktop sidebar;
- mobile drawer;
- topbar quick actions.

Правила:

- logout должен быть POST-form;
- скрытая mobile navigation должна быть доступна с клавиатуры.

### Pagination

Назначение: навигация по длинным спискам.

Где используется: expenses, incomes, items, projects, reports при необходимости.

Внешний вид:

- compact pagination внизу списка;
- previous/next;
- текущая страница выделена.

Состояния:

- default;
- hover;
- focus;
- disabled for unavailable prev/next;
- active current page.

Варианты:

- page numbers;
- simple prev/next для mobile.

Правила:

- фильтры должны сохраняться при переходе страниц;
- на mobile не показывать слишком много номеров страниц.

### Filters/Search

Назначение: сузить длинные списки и отчеты.

Где используется: tasks, projects, finance lists, reports.

Внешний вид:

- toolbar над списком;
- search input;
- select filters;
- date range/month filter;
- reset button.

Состояния:

- no filters;
- filters applied;
- loading results;
- no results.

Варианты:

- compact inline filters на desktop;
- collapsible filter panel на mobile.

Правила:

- показывать, что фильтры применены;
- reset должен возвращать список к базовому состоянию;
- no results отличается от truly empty state.

### Empty State

Назначение: объяснить отсутствие данных и предложить действие.

Где используется: все списки, dashboard blocks, reports.

Внешний вид:

- короткий заголовок;
- пояснение;
- primary action, если уместно;
- без декоративных иллюстраций как обязательного элемента.

Состояния:

- no data ever;
- no results after filters;
- no access.

Варианты:

- page empty;
- section empty;
- table empty.

Правила:

- empty state должен быть конкретным для раздела;
- не писать технические сообщения вроде "queryset пуст".

### Loading Skeleton

Назначение: показать загрузку без дергания layout.

Где используется: dashboard metrics, tables, cards, dynamic filters.

Внешний вид:

- нейтральные серые блоки;
- размеры совпадают с будущим контентом;
- мягкая shimmer-анимация допустима, но не обязательна.

Состояния:

- loading;
- loaded;
- error after loading.

Варианты:

- metric skeleton;
- table row skeleton;
- card skeleton;
- form submit loading.

Правила:

- skeleton не должен менять layout после загрузки;
- не использовать бесконечный spinner на весь экран для обычных server-rendered переходов.

## 7. Адаптивность

### Большой desktop: 1440px и шире

- Sidebar открыт постоянно.
- Контент может использовать 2-3 колонки на dashboard.
- Таблицы отображаются полностью.
- Фильтры показываются inline toolbar.
- Page header содержит заголовок, breadcrumbs и quick actions в одну строку.
- Формы остаются ограниченными по ширине, не растягиваются на весь экран без необходимости.

### Обычный ноутбук: 1366px

- Sidebar открыт.
- Dashboard использует 2 колонки для основных блоков.
- Metric cards могут быть в 3-4 колонки.
- Таблицы должны помещаться без горизонтального scroll для основных колонок.
- Secondary columns можно скрывать или переносить в detail/overflow, если данных много.

### Маленький ноутбук: 1024px

- Sidebar может стать уже или перейти в collapsed mode.
- Dashboard переходит к 2 колонкам или одной широкой колонке для сложных блоков.
- Фильтры могут переноситься на вторую строку.
- Табличные action buttons желательно заменить на компактное меню.
- Контентные отступы уменьшаются с 24-32px до 16-24px.

### Планшет: 768px

- Sidebar скрыт, используется mobile header и drawer.
- Контент в одну колонку.
- Metric cards в 2 колонки, если хватает места, иначе в одну.
- Формы в одну колонку.
- Таблицы CRUD-списков превращаются в карточки строк.
- Фильтры сворачиваются в collapsible panel или показываются блоком над списком.
- Primary action остается в page header или становится full-width button под заголовком.

### Телефон: 390px

- Минимальная ширина поддержки: 360-390px.
- Все формы в одну колонку.
- Кнопки в формах full-width или stacked.
- Touch target минимум 44px.
- Page padding 12-16px.
- Sidebar заменен drawer или bottom navigation.
- Длинные названия задач, проектов и товаров переносятся, не выходят за контейнер.
- Табличные строки показываются карточками: label + value.
- Sticky actions допустимы только если не перекрывают поля, системную клавиатуру и footer.

### Общие правила перестроения

- Блоки dashboard перестраиваются из 3-4 колонок в 2, затем в 1.
- Фильтры на desktop горизонтальные, на mobile вертикальные или раскрывающиеся.
- Таблицы операций на mobile становятся карточками.
- Аналитические таблицы могут иметь горизонтальный scroll, если это явно лучше карточек.
- Sidebar скрывается ниже 1024px или 768px в зависимости от финального app shell.
- Списки задач сохраняют быстрый доступ к checkbox, а вторичные действия уходят в overflow.
- Модалки на телефоне занимают почти всю ширину экрана, но сохраняют отступы 16px.

## 8. Дизайн для будущих модулей

Дизайн должен быть расширяемым. NextDo планируется развивать от списка задач к приложению, где проекты связывают задачи, доходы, расходы, товары/объекты и аналитику.

### Расширение меню

Sidebar должен поддерживать добавление новых разделов без перестройки:

- Dashboard;
- Задачи;
- Проекты;
- Финансы;
- Отчеты;
- Настройки.

Финансовый раздел должен поддерживать вложенные пункты или внутренние tabs:

- Обзор;
- Расходы;
- Доходы;
- Товары/объекты;
- Категории;
- Отчеты.

Если будущих пунктов станет больше, sidebar должен поддерживать группировку по секциям.

### Добавление новых страниц

Каждая новая страница должна использовать один из базовых layout-паттернов:

- dashboard page;
- list CRUD page;
- create/edit form page;
- detail page;
- report page;
- auth page.

Это позволит добавлять новые сущности без изобретения нового дизайна для каждой страницы.

### Финансовый модуль

Финансовый дизайн должен учитывать:

- одну валюту v1: RUB;
- хранение сумм в копейках на backend, но отображение пользователю в рублях;
- доходы и расходы как разные типы операций;
- расход с несколькими строками;
- товар/объект как справочник;
- категории доходов и расходов;
- фильтры по дате, месяцу, проекту, категории, товару/объекту;
- месячную статистику.

### Статистика и аналитика

Аналитические блоки должны быть единообразны:

- metric card для ключевого числа;
- compact table для breakdown;
- chart только если данные действительно становятся понятнее;
- фильтры всегда рядом с отчетом;
- выбранный период явно виден.

### Связь сущностей

Интерфейс должен показывать связи без перегрузки:

- задача может иметь проект;
- доход или расход может иметь проект;
- расход состоит из товаров/объектов;
- категории помогают группировать доходы и расходы;
- dashboard связывает все это в краткой сводке.

Связанные сущности показывать через badges, links и secondary text, а не через длинные технические подписи.

### Единый стиль CRUD-разделов

Для проектов, финансовых категорий, товаров/объектов, доходов и расходов использовать один паттерн:

- page header;
- primary action;
- filters/search;
- list/table;
- row actions;
- pagination;
- empty/loading/error states;
- create/edit form;
- confirm delete.

## 9. UX-правила

### Формы

- Каждое поле имеет label.
- Placeholder используется только как пример ввода.
- Обязательные поля отмечаются аккуратно и объясняются при ошибке.
- При ошибке отправки введенные данные сохраняются.
- Ошибки показываются рядом с полями.
- Общая ошибка формы показывается сверху формы.
- Submit button переходит в loading state после отправки.
- Повторная отправка формы блокируется.
- Длинные формы делятся на логические секции.
- В финансовых формах итог суммы должен обновляться понятно и находиться рядом с действием сохранения.

### Ошибки

- Ошибки должны быть написаны человеческим языком.
- Не показывать traceback или технические исключения пользователю.
- Ошибка доступа отличается от ошибки валидации.
- Если данные не найдены из-за фильтров, показывать no results, а не error.
- Если действие не выполнено, уведомление должно объяснить, что можно сделать дальше.

### Опасные действия

- Удаление задач, проектов, доходов, расходов и справочников требует подтверждения.
- Кнопка подтверждения удаления использует danger style.
- В тексте подтверждения указывать объект: "Удалить задачу '...'?".
- Для архивирования проекта использовать более мягкий warning/secondary pattern, если данные сохраняются.
- Если объект связан с историей финансов, предпочтительно архивирование вместо физического удаления, если это будет предусмотрено бизнес-логикой.

### Успешные действия

- Создание, сохранение, удаление и архивирование подтверждаются toast/message.
- Сообщение короткое: "Проект создан", "Расход сохранен", "Задача удалена".
- Success-сообщение не должно блокировать дальнейшую работу.
- После создания из формы пользователь возвращается к релевантному списку или detail-странице.

### Редактирование

- Edit actions должны быть доступны в row actions или detail page.
- На форме редактирования явно показывать, какой объект редактируется.
- Если пользователь уходит с несохраненными изменениями, в будущем можно добавить предупреждение.
- Primary action на edit form - "Сохранить", secondary - "Отменить".

### Длинные списки

- Использовать pagination для больших списков.
- Добавить search, если список может быть длиннее 20-30 элементов.
- Фильтры должны сохраняться при pagination.
- Сортировка должна быть визуально понятной.
- На mobile не перегружать строки действиями: вторичные действия переносить в overflow.

### Фильтрация и поиск

- Фильтры располагаются над списком.
- Пользователь видит, какие фильтры применены.
- Есть действие "Сбросить".
- No results state отличается от empty state.
- Для финансовых отчетов выбранный период показывается явно.

### Как избегать перегрузки

- Не показывать все возможные действия одновременно.
- Primary action должен быть один.
- Вторичные действия группировать.
- Подробности показывать по переходу, раскрытию или в detail-view.
- В dashboard показывать только summary и ссылки на подробности.
- Не использовать декоративные блоки вместо полезной информации.

## 10. Технические требования к реализации дизайна

- Не ломать существующую бизнес-логику задач и авторизации.
- Сохранять server-rendered Django templates как основной frontend-подход.
- Разделить UI и бизнес-логику: расчеты, фильтрация и доступ не должны находиться в шаблонах.
- Вынести общие стили в static CSS, например `static/css/app.css`, когда начнется реализация.
- Не дублировать стили кнопок, полей, карточек и таблиц в каждом шаблоне.
- Создать переиспользуемые template partials или include-компоненты для повторяющихся блоков, если это соответствует структуре Django templates.
- Соблюдать единый стиль именования CSS-классов.
- Не делать хаотичные CSS-правки внутри отдельных страниц.
- Не внедрять тяжелый frontend framework без отдельного решения.
- Для действий изменения данных использовать POST и CSRF.
- Logout должен оставаться POST-действием.
- Удаление должно иметь подтверждение.
- Проверять доступность: keyboard navigation, focus-visible, контраст, labels, `aria-label` для icon-only actions.
- Проверять производительность: не добавлять тяжелые библиотеки ради простых UI-сценариев.
- Избегать больших несвязанных изменений: app shell, компоненты, задачи, финансы и адаптивность реализовывать этапами.
- Существующие тесты задач не должны ломаться без осознанного обновления под новые POST-сценарии.
- Новые CRUD-разделы должны иметь тесты доступа пользователя к своим данным.
- Тексты интерфейса должны быть на русском языке и не смешивать terminal-стилистику с рабочим UI.
