# План развития NextDo: редизайн, финансы, HTTPS

Дата анализа: 2026-05-19

## 1. Результат анализа проекта

### Текущее устройство

- Проект: небольшое Django-приложение `NextDo` для личных задач.
- Стек: Python 3.13, Django 6.0.3, PostgreSQL, Gunicorn, Nginx, Docker Compose, `python-decouple`.
- Архитектура: классический Django monolith без REST API, frontend собран на Django templates.
- Основные директории:
  - `src/config/` - настройки, корневые URL, WSGI/ASGI.
  - `src/task/` - модель `Task`, формы, views, urls, tests.
  - `src/templates/` - `base.html`, login/logout, список задач.
  - корень проекта - `Dockerfile`, `docker-compose.yml`, `nginx.conf`, `.env.example`, `README.md`.
- Документация: есть `README.md`; папки `docs/` до этого плана не было.
- Дизайн-файл: есть `design.pen`, не отслеживается Git. Через Pencil видно, что документ почти пустой: один frame 800x600, переменных и компонентов нет.

### Текущий функционал задач

- Есть одна доменная модель `Task`:
  - `title`;
  - `user`;
  - `is_completed`;
  - `created_at`.
- Все задачи привязаны к пользователю.
- Авторизация есть через стандартную Django auth.
- Публичной регистрации пользователей нет.
- Страница `/tasks/` одновременно показывает список задач и форму создания.
- Завершенные задачи скрываются на клиенте через небольшой inline JavaScript.
- Toggle и delete сейчас выполняются через GET-ссылки. Это работает, но для изменения данных лучше перейти на POST.

### Маршруты

- `/login/` - вход.
- `/logout/` - выход.
- `/tasks/` - список и создание задач.
- `/tasks/<task_id>/toggle/` - переключение статуса задачи.
- `/tasks/<task_id>/delete/` - удаление задачи.
- `/admin/` - Django admin.
- `handler404` редиректит неизвестные URL на `/tasks/`.

### Деплой и конфигурация

- Есть Docker Compose с сервисами `db`, `web`, `nginx`.
- `web` запускает миграции, `collectstatic`, затем Gunicorn на `8000`.
- `nginx` слушает только `80:80`.
- HTTPS, сертификаты, 443-порт и автопродление не настроены.
- Секреты должны приходить из `.env`; `.env.example` содержит только шаблон.

### Тесты

- Есть тесты модели и views для задач.
- Локально команда `python manage.py test` не запустилась, потому что в активном Python-окружении не установлен Django.
- В плане реализации нужно проверять тесты через подготовленный virtualenv или через Docker.

### Источник истины

- `README.md` в целом совпадает с кодом и может считаться актуальной документацией.
- При расхождениях между README и кодом источником истины считать код, а README обновлять в рамках реализации.

## 2. Принятые решения и открытые вопросы

### Принятые решения

- Финансовые данные персональные: каждый пользователь видит только свои проекты, доходы, расходы, товары/объекты и категории.
- Добавляется новая сущность `Project`.
- `Project` нужно связать и с финансами, и с задачами, чтобы проект стал общей доменной осью.
- HTTPS планируется для VPS + Docker.
- Базовый вариант HTTPS: Let's Encrypt, Nginx в Docker, редирект HTTP -> HTTPS, автоматическое продление.
- Новая визуальная концепция: рабочий dashboard, а не декоративный landing.
- Финансовый v1 включает категории и месячную статистику.
- Валюта v1: одна валюта `RUB`.
- Денежные суммы хранить целыми числами в копейках, а не `float`.

### Вопросы, которые нужно подтвердить перед реализацией

- Есть ли уже домен для production и какое имя использовать в `server_name`.
- Где физически будет VPS и кто управляет DNS.
- Нужно ли хранить сертификаты в Docker volume или использовать certbot/nginx companion.
- Нужно ли оставлять доступ по `localhost` для локальной разработки без HTTPS.
- Как в интерфейсе называть товары/объекты: `Товары`, `Объекты`, `Позиции`, `Активы` или другое название.
- Нужна ли публичная регистрация пользователей или пользователей создает администратор.
- Нужно ли редактирование задач в рамках редизайна или оставить задачу только create/toggle/delete.

## 4. Архитектура финансового модуля

### Новые Django apps

- `projects` - общая сущность проектов.
- `finance` - доходы, расходы, товары/объекты, категории и статистика.

### Модели данных

- `Project`
  - `user` - владелец;
  - `name` - обязательное название;
  - `description` - опционально;
  - `is_archived` - для скрытия старых проектов;
  - `created_at`, `updated_at`.
- `Task`
  - добавить `project`, nullable FK на `Project`;
  - сохранить обратную совместимость для существующих задач без проекта.
- `FinanceCategory`
  - `user`;
  - `name`;
  - `type`: `income` или `expense`;
  - уникальность имени в рамках пользователя и типа.
- `FinanceItem`
  - `user`;
  - `name`;
  - `description` опционально;
  - используется как товар/объект/позиция расхода.
- `Expense`
  - `user`;
  - `project` nullable;
  - `category` nullable или required после согласования;
  - `occurred_on`;
  - `note`;
  - `created_at`, `updated_at`.
- `ExpenseLine`
  - `expense`;
  - `item`;
  - `quantity` как `DecimalField`;
  - `unit_price_cents` как `PositiveIntegerField`;
  - итог строки считается как `quantity * unit_price_cents` с округлением до копеек.
- `Income`
  - `user`;
  - `project` nullable;
  - `category` nullable или required после согласования;
  - `amount_cents`;
  - `source`;
  - `occurred_on`;
  - `note`;
  - `created_at`, `updated_at`.

### Бизнес-логика

- Все queryset обязаны фильтроваться по `request.user`.
- Проверять, что выбранные `Project`, `FinanceCategory` и `FinanceItem` принадлежат текущему пользователю.
- Суммы не хранить в `float`.
- Общая сумма расхода не вводится вручную, а считается по строкам `ExpenseLine`.
- Расчеты статистики вынести из templates во view/service/query layer.
- Для v1 использовать RUB без обменных курсов.

### Страницы

- `/dashboard/` - общий dashboard.
- `/tasks/` - обновленный список задач.
- `/projects/` - список и создание проектов.
- `/finance/` - обзор финансов.
- `/finance/expenses/` и `/finance/expenses/new/` - расходы.
- `/finance/incomes/` и `/finance/incomes/new/` - доходы.
- `/finance/items/` - товары/объекты.
- `/finance/reports/` - статистика.

### Статистика и фильтры

- Общая статистика: доходы, расходы, баланс за выбранный период.
- По проектам: доходы, расходы, баланс.
- По товарам/объектам: сумма расходов, количество, частота покупок.
- По категориям: доходы/расходы за период.
- По месяцам: группировка через `occurred_on`.
- Фильтры:
  - дата от/до;
  - месяц;
  - проект;
  - категория;
  - товар/объект.

### Идеи не для обязательного v1

- Повторяющиеся расходы.
- Счета/кошельки.
- Мультивалютность.
- CSV export/import.
- План/факт.
- Прогноз регулярных трат.
- Графики повышенной сложности.
- Иерархия объектов, например `машина -> топливо / запчасти / обслуживание`.

## 5. HTTPS и production-конфигурация

### Целевая схема

- VPS с Docker Compose.
- Nginx принимает `80` и `443`.
- HTTP всегда редиректится на HTTPS.
- Сертификаты выпускаются через Let's Encrypt.
- Сертификаты и challenge-файлы хранятся в Docker volumes.
- Секреты не попадают в репозиторий.

### Изменения

- Добавить production-ориентированный Nginx config:
  - `server_name` из домена;
  - `/.well-known/acme-challenge/`;
  - SSL certificate paths;
  - redirect 80 -> 443;
  - security headers.
- Обновить `docker-compose.yml` или добавить отдельный `docker-compose.prod.yml`.
- Добавить certbot/acme сервис или companion-контейнер.
- Обновить `.env.example`:
  - `ALLOWED_HOSTS`;
  - `CSRF_TRUSTED_ORIGINS`;
  - production `DEBUG=False`;
  - домен/email для Let's Encrypt без секретов.
- В Django settings добавить production flags:
  - `SECURE_PROXY_SSL_HEADER`;
  - `SESSION_COOKIE_SECURE`;
  - `CSRF_COOKIE_SECURE`;
  - `SECURE_SSL_REDIRECT` только для production;
  - корректные `CSRF_TRUSTED_ORIGINS`.

### Риски HTTPS

- Нельзя выпустить Let's Encrypt без реального домена и DNS на VPS.
- Нельзя хранить private key сертификата в Git.
- Нужно аккуратно разделить local и production config, чтобы локальная разработка не ломалась.
- Нужно проверить proxy headers, иначе Django может неверно определять HTTPS.

## 7. Риски

- Расширение доменной модели может сломать существующие задачи, если миграции не сделать nullable/backward-compatible.
- Финансовые расчеты могут стать неточными, если использовать `float` или смешать рубли и копейки.
- Неправильная фильтрация по пользователю приведет к утечке чужих финансовых данных.
- Inline CSS/JS в текущих шаблонах усложнит редизайн, если не вынести общие стили.
- HTTPS нельзя полностью проверить без production-домена.
- Автоматическое продление сертификатов требует проверки на реальном VPS.
- Текущие локальные тесты требуют подготовленного Python-окружения или запуска через Docker.

## 8. Критерии проверки

### Функциональные сценарии

- Пользователь видит только свои задачи, проекты и финансы.
- Пользователь создает проект и привязывает к нему задачу.
- Пользователь создает расход с новым или существующим товаром/объектом.
- Итог расхода считается по строкам.
- Пользователь создает доход с источником и проектом.
- Статистика корректно считается по датам, проектам, категориям и товарам/объектам.
- Удаление и изменение данных выполняются через POST с CSRF.

### Технические проверки

- `python manage.py test` проходит в подготовленном окружении или Docker.
- Миграции применяются на чистую базу.
- `docker compose up --build` поднимает локальный проект.
- Production compose/nginx config не требует секретов в Git.
- HTTP редиректится на HTTPS на production-домене.
- Cookies и CSRF работают под HTTPS.

### UI-проверки

- Новый дизайн не похож на текущий terminal/cyberpunk UI.
- Desktop dashboard удобен для задач и финансов.
- Mobile layout не ломает формы, таблицы и карточки.
- Empty/error/loading states предусмотрены.
- Тексты не выходят за контейнеры.
